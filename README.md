# sarun

super easy controlled execution for linux. this is not containers. this is for running normal programs directly, but not really.
two parts - "ui"/"server", and "sandbox"/"client". both are exact same command. 'sarun' with just options starts ui, 'sarun <cmd>' runs cmd under sandbox.
sandbox uses bwrap to run command in a copy-on-write overlay covering the whole filesystem, so that all file modifications by the program get captured in that overlay's upper layer. (except of course /proc, /dev and such that make no sense whatsoever to cover with overlay, and they instead are mounted by bwrap)
that overlay is now served by the UI itself: the UI owns ONE pyfuse3 mount (created the ordinary way via the setuid fusermount3 helper — no root, no user namespace, no fd-passing, no inner mount), multiplexed so each running sandbox is a subfolder `<mnt>/<sid>` presenting a merged view of lower=/ plus that session's private upper (under the runtime dir as live/<sid>/up + live/<sid>/index.db). the runner then does the exact same bwrap as before, only binding `<mnt>/<sid>` as / instead of a per-box fuse-overlayfs mount — every other bwrap flag (own pid/ipc/uts/net ns, all caps dropped, runs as you) is unchanged. because the UI sits in the filesystem I/O path it gets live feedback (the Changes view updates as the box writes) and an explicit per-write policy hook, plus writer provenance (pid/exe/argv) recorded per change. consequence: `sarun <cmd>` now REQUIRES the UI to be running, and fails fast if it isn't.
sandbox removes direct network connectivity and uses mitmproxy to provide network access, with all requests either being approved or rejected by the matching rule from the ruleset, or, if no rule matches, by user action (hence need for server and ui. sandboxes cannot directly ask user. what if sandbox runs from a script? but that ui can. so sandbox connects to that ui on startup), that can accept or reject or create a rule for duration of session, or permanently (that is added to ~/.config/sarun/netrul.txt).
to make life of mitmproxy easier, the overlay synthesizes the system certificate stores with the mitmproxy certificate appended as trusted — as *virtual* files (no on-disk artifact in either layer), so the box trusts the re-signing CA but the modified bundles never show up as reviewable changes.
sandbox, by virtue of how bwrap works, prevents program from messing with system by running sudo, writing to all kinds of sockets, or messing with other privileged actions.
the captured upper for each session lives under the runtime dir (live/<sid>) while running, then consolidates on exit into a patch (text changes) + sqlar archive (binary/symlink/deletions, with provenance) under the state dir, and is visible and browsable in the UI that lists all sessions. upon completion (or a crash) of a sandbox session the ui provides visual hint that these changes are safe to inspect and merge into host filesystem as is, or edit before merging, or merge just some bits, or delete. this is facilitated by the interactive nature of the ui, support for deletions/whiteouts (tracked in the per-session index, not on-disk markers), automatic diff view for textual files and easy to use "smalltalk-like" file browser with file/diff information in the large bottom area, and folder path in the string of navigation panes above.
while sandbox has not terminated, UI lets inspect it's process tree (similar to how htop does) and intervene by poking the processes. it is also possible to destroy whole sandbox session.
all sandbox processes run as the user that started the sandbox. again, this is not a container.
